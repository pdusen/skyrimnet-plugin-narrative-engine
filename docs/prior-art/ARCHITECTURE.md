## 1. High-Level Overview

IntelEngine gives Skyrim NPCs the ability to autonomously perform complex, multi-step tasks in the game world. It
operates in three major modes:

### Player-Driven Actions

Players tell NPCs to do things through natural dialogue (powered by SkyrimNet's LLM). NPCs can:

- **Travel** to any named or semantic location ("go to the Bannered Mare", "go upstairs")
- **Fetch** another NPC and bring them to the player
- **Deliver messages** to other NPCs, optionally scheduling a follow-up meeting
- **Escort** an NPC to a destination
- **Search** for a missing person (player-led exploration)
- **Schedule** any of the above for a specific future game time ("meet me at the inn at dusk")

### Autonomous Story Engine (Dungeon Master)

An LLM periodically evaluates world state and decides whether to dispatch NPCs on story-driven errands:

- NPCs seek out the player after long absences
- Informants arrive with gossip traceable through real memory chains
- Ambushes and stalkers create tension from grudges
- Quest-givers send the player on combat/rescue/find-item missions
- NPCs interact with each other off-screen (arguments, deals, celebrations)

### Faction Politics

A political simulation runs in parallel:

- 9 configurable factions with rivalries, alliances, and war thresholds
- A "Political DM" (LLM) generates faction events every ~6 game hours
- Wars can escalate from diplomatic incidents to full-scale battles
- Player-witnessed battles spawn soldiers with morale tracking and wave reinforcements
- Player standing with factions shifts based on conduct, battle participation, and NPC reports

│  └──────┘ └───────┘ └──────────┘ └────────┘ └──────────────┘  │
└────────────────────────────┬────────────────────────────────────┘

**Why two languages?** C++ handles anything performance-sensitive or requiring direct engine access (index building,
fuzzy search, SQLite, threading, 150ms polling). Papyrus handles anything requiring game-level orchestration (AI
packages, quest aliases, ModEvents, StorageUtil persistence). The boundary is the `Papyrus.cpp` bridge with 200+ native
functions.

## 5. C++ Native Plugin (SKSE DLL)

### 5.1 Plugin Entry Point & Lifecycle

**Files:** `Plugin.cpp`, `Plugin.h`

The plugin registers with SKSE via `SKSEPluginLoad()` and handles these lifecycle events:

- **`kDataLoaded`** — Build NPC/Item/Location indexes; load Settings; initialize SkyrimNet API; register
  Papyrus native functions; register SkyrimNet decorators.
- **`kNewGame`** — Generate per-save unique ID; initialize PoliticalDB.
- **`kPostLoadGame`** — Generate/restore per-save ID; initialize PoliticalDB with timeline cleanup; resolve
  quest handle; fixup stale script properties; dispatch `Maintenance()` to Papyrus.
- **`kPreLoadGame`** — Reset SlotTracker, BattleManager, StuckDetector, DepartureDetector, OffScreenTracker.

**Per-Save Isolation:** Each save gets a unique ID (`timestamp_randomSuffix`) used to create an isolated SQLite
database (`IntelEngine-{saveID}.db`). This allows save-scumming without corrupting political history — future events
(from a timeline the player reverted from) are deleted and standings are recalculated from the remaining chronological
history.

**Script Property Fixup:** On existing saves, `FixupScriptProperties()` patches stale cross-references between Papyrus
scripts (e.g., if `IntelEngine_Core` has a reference to `IntelEngine_Travel` that points to an old object). This is
done via direct Papyrus VM property manipulation.

### 5.2 Papyrus Native Bridge

**Files:** `Papyrus.cpp`, `Papyrus.h`

This is the largest file in the project (~5000+ lines) and serves as the sole interface between C++ and Papyrus. It
registers ~200 native functions organized into these domains:

- **NPC Search** — `FindNPCByName()`, `FindNPCByNameNear()`, `GetNPCNameSuggestion()`, `IsNPCAccessible()`
- **Location Resolution** — `ResolveAnyDestination()`, `ResolveSemanticLocation()`, `GetAvailableSemanticDirections()`
- **Action Validation** — `ValidateAction()`, `GetActionFailureReason()`
- **Cell Analysis** — `GetCellDoors()`, `GetCellSpatialInfo()`, `IsDoorExterior()`, `IsDoorUpward()`
- **Item Search** — `FindItemInInventory()`, `FindNearbyItemByName()`
- **Distance/Math** — `GetDistance3D()`, `CalculateDeadlineFromDistance()`, `GetOffsetBehind()`
- **Time Parsing** — `ParseTimeCondition()`, `CalculateTargetGameTime()`
- **Detection Polling** — `CheckDepartureStatus()`, `CheckStuckStatus()`, `CheckOffScreenProgress()`
- **Proximity** — `ArmProximityArrival()`, `DisarmProximityArrival()`
- **Slot Management** — `UpdateSlotState()`, `ClearSlotState()`, `SyncArraysFromSlotTracker()`
- **Story Engine** — `GetRandomStoryCandidate()`, `GetMemoryDrivenCandidate()`, `BuildDungeonMasterContext()`
- **Quest Spawning** — `SpawnQuestEnemies()`, `SpawnQuestChest()`, `FindDeeperSpawnPoint()`
- **MemoryDB** — `GetNPCMemories()`, `GetRecentWorldEvents()`, `GetNPCRelationshipSummary()`
- **Faction Politics** — `GetFactionRelation()`, `BuildPoliticalContext()`, `DeclareWar()`, `ProcessWarTick()`
- **Battle** — `StartBattle()`, `PollBattleState()`, `ExecuteFullBattleSpawn()`, `FinalizeBattle()`
- **Dashboard** — `PushDashboardFullState()`, `NotifyDashboardSlotChanged()`, `ReloadDashboardUI()`
- **Dialogue Safety Net** — `RunSafetyNetCheck()`, `HasScheduleKeywords()`, `BuildSafetyNetContextJson()`
- **String Utilities** — `StringToLower()`, `LevenshteinDistance()`, `StringEscapeJson()`

Every function is registered as a global native on the `IntelEngine` script class, callable from any Papyrus script.

### 5.3 NPC Index

**Files:** `NPCIndex.cpp`, `NPCIndex.h`

A singleton that indexes all loaded NPCs for fast lookup and serves as the story candidate selection engine.

**Index Building:**

- On `kDataLoaded`, iterates all four process list tiers (high/middleHigh/middleLow/low) to index every loaded actor
- Maintains three maps: `name → Actor*` (loaded only), `name → FormID` (all NPCs), `name → location name`
- Can be refreshed at runtime for dynamic changes

**Search Cascade:**

1. Exact case-insensitive match
2. Article-stripped match ("the Bannered Mare" → "Bannered Mare")
3. Token overlap scoring (word-by-word comparison)
4. Levenshtein distance (character-level typo tolerance)

**Story Candidate Selection:**

- `GetRandomStoryCandidate()` — Random eligible NPC (alive, not disabled, not in combat, not in player's cell, no
  active task, no cooldown)
- `GetMemoryDrivenCandidate()` — Ranked by SkyrimNet engagement score (NPCs the player has interacted with most)
- `GetRelatedCandidate()` — NPC related to a given actor (shared factions, location, household)
- `FindMessengerForSender()` — Cascade: household member → social associate → guard → any civilian (for high-status
  NPCs who can't travel personally)

**Archetype Classification:**
NPCs are classified as WARRIOR, MAGE, ROGUE, PRIEST, NOBLE, BARD, or CIVILIAN based on faction membership and class.
This determines which story types they're eligible for (e.g., only warriors/rogues can ambush; only civilians/nobles
send messengers).

**Dispatch History:**
A ring buffer of the last 6 story dispatches (type, subtype, NPC name, narration, outcome) is maintained and injected
into the DM prompt so the LLM can see its own recent decisions, avoid repetition, and learn from rejections.

**Policy Enforcement:**
MCM-configurable policies control candidate filtering:

- Danger zone policy (0–3): controls which NPCs can be dispatched to dangerous locations
- Player home policy (0–3): controls dispatch into player homes
- Hold restriction policy (per story type): restricts to same hold/settlement

### 5.4 Location Resolver

**Files:** `LocationResolver.cpp`, `LocationResolver.h`

Resolves human-readable location names and semantic directions to actual game cells/locations.

**Named Location Resolution:**

- Indexes all `TESObjectCELL` and `BGSLocation` forms at startup
- Fuzzy matches player/NPC-provided location names against the full index
- Falls back from cell → BGSLocation → parent location → name extraction from doors

**Semantic Direction Resolution:**
Handles terms like "upstairs", "outside", "back room", "cellar", "home":

- `ResolveUpstairs()`/`ResolveDownstairs()` — Scans doors by Z-coordinate delta + name keywords ("upper", "cellar") +
  furniture scanning
- `ResolveOutside()`/`ResolveInside()` — Door type analysis (interior ↔ exterior transitions)
- `ResolveHome()` — Resolves NPC/player/named home via pre-built home index (maps NPCs to their owned beds/cells)
- `ResolveBedroom()`/`ResolveKitchen()` — Furniture-based scanning (beds, cooking stations)
- `ResolveWater()` — Interior: bath/wash furniture; Exterior: river/swim proximity

**Unified Resolver:**
`ResolveAnyDestination()` tries semantic → door scanning → parent location → named resolution in sequence, returning
the first successful match. This is the primary entry point used by travel dispatch.

**Home Door Access:**
For NPCs traveling to homes, the system temporarily unlocks the target's home door and marks it public (preventing
trespass bounties), then re-locks it after the task completes.

### 5.5 Item Index

**Files:** `ItemIndex.cpp`, `ItemIndex.h`

Indexes all weapons, armor, misc items, and spell tomes for quest reward spawning. Maintains a filtered list of
"valuable" items (above a configurable gold threshold) for random quest reward selection. Supports fuzzy name matching
and rotation tracking (recently-used items are excluded from random selection).

### 5.6 Slot Tracker

**Files:** `SlotTracker.cpp`, `SlotTracker.h`

**The authoritative source of truth for task state.** C++ owns the canonical task data, not Papyrus's StorageUtil.

- Mirrors 5 concurrent task slots (matching the 5 quest alias pairs in the ESP)
- Each slot tracks: agent FormID, state, task type, target name, destination, speed, deadline, cooldown, off-screen
  arrival flag
- Persisted via SKSE co-save serialization (survives save/load)
- On game load, `SyncArraysFromSlotTracker()` pushes C++ state back to Papyrus arrays, avoiding StorageUtil
  deserialization latency
- `BuildBusyReason()` provides a single-source-of-truth formatted string for SkyrimNet's busy status decorator

### 5.7 Action Validator

**Files:** `ActionValidator.cpp`, `ActionValidator.h`

Pre-validates actions before dispatch to prevent the LLM from hallucinating impossible tasks:

- `ValidateTravel()` — Is the destination resolvable?
- `ValidateFetchNPC()` — Does the NPC exist? Is it accessible?
- `ValidateDeliverMessage()` — Can the recipient be found?
- Returns specific failure codes: `"no_destination"`, `"npc_not_found"`, `"npc_dead"`, `"npc_is_self"`, etc.

### 5.8 Detection Systems

Three complementary systems monitor NPC travel progress:

**DepartureDetector** (`DepartureDetector.cpp/h`):

- Tracks whether an NPC has actually started moving after receiving a travel command
- Baseline position set when travel begins
- Returns: 0=too early to check, 1=departed, 2=soft recovery needed, 3=escalate

**StuckDetector** (`StuckDetector.cpp/h`):

- Detects NPCs stuck on pathfinding obstacles during travel
- Three-level escalation: 0=moving normally, 1=soft recovery (re-evaluate AI packages), 3=teleport
- Progressive teleport distances: 2000 → 1000 → 500 → 250 units closer to target on each escalation
- Configurable max checks and max recovery attempts via INI settings

**OffScreenTracker** (`OffScreenTracker.cpp/h`):

- Tracks travel progress for NPCs in unloaded cells (where position polling doesn't work)
- Two-phase: before estimated arrival (update baseline by game time), after arrival time passes (check if NPC has
  stopped moving)

### 5.9 Proximity Monitor

**Files:** `ProximityMonitor.cpp`, `ProximityMonitor.h`

Replaces Papyrus's 3-second `OnUpdate` poll with a 150ms C++ worker thread for arrival detection. When the player
approaches a task destination, this detects arrival ~20x faster than Papyrus could.

- `Arm(slot)` — Start watching a slot for arrival (distance threshold: 150 units for actors, 300 for markers)
- Checks both 2D distance and Z-axis tolerance (120u for actors, 200u for markers)
- Arrival callbacks are marshaled back to the main thread via `AsyncDispatch` to maintain Papyrus VM thread safety

### 5.10 Async Dispatch

**Files:** `AsyncDispatch.cpp`, `AsyncDispatch.h`

Manages a dedicated worker thread for the three-phase async pattern used throughout the plugin:

- **Phase A** (main thread): Snapshot engine state into plain value structs (no dangling pointers)
- **Phase B** (worker thread): Perform SQL queries, markdown formatting, context building
- **Phase C** (main thread): Dispatch the result to Papyrus via `DispatchMethodCall`

This architecture keeps SQL/IO work off the game thread while ensuring all engine API calls and Papyrus dispatch happen
on the main thread (as required by the engine).

### 5.11 Memory DB

**Files:** `MemoryDB.cpp`, `MemoryDB.h`

A thin client layer over SkyrimNet's memory API. Queries are dispatched via C++ function pointers loaded from
`SkyrimNet.dll` at runtime:

- `GetNPCMemories()` — Formatted memories for a specific NPC
- `GetRecentWorldEvents()` — Recent world events
- `GetActiveStoryNPCs()` — NPCs ranked by story engagement
- `GetNPCRelationshipSummary()` — Relationship between two NPCs
- `GetPlayerInteractionCount()` — How often the player has interacted with an NPC

### 5.12 Political DB

**Files:** `PoliticalDB.cpp`, `PoliticalDB.h`

An embedded SQLite database (`IntelEngine-{saveID}.db`) that persists all faction politics data:

| Table                     | Purpose                                                       |
| ------------------------- | ------------------------------------------------------------- |
| `faction_relations`       | Pairwise faction relation scores (-100 to +100)               |
| `faction_events`          | Chronological political event log                             |
| `player_faction_standing` | Player standing per faction                                   |
| `faction_wars`            | Active and historical wars (morale, strength, battles fought) |
| `battles`                 | Off-screen battle results                                     |

**Save-Scum Recovery:** On game load, `CleanupFutureEvents()` deletes any events with timestamps beyond the current
game time, then `RecalculatePlayerStandings()` replays the remaining history chronologically to reconstruct accurate
standings.

### 5.13 Faction Politics

**Files:** `FactionPolitics.cpp`, `FactionPolitics.h`, `FactionConfigLoader.cpp/h`

The political system singleton that orchestrates faction interactions:

**Configuration (factions.yaml):**
Each faction defines: display name, type (military/guild/political), hold, Skyrim faction FormID, leader NPCs, rivals,
allies, army strength, war threshold, conflict style (brawl/sabotage/assassination/proxy/military), soldier leveled
list template, and prison location.

Default faction-pair relations are pre-seeded (e.g., Imperials +50 with Thalmor, -30 with Stormcloaks).

**Political DM Tick:**
Every ~6 game hours, the system:

1. Snapshots engine state (Phase A)
2. Builds political context with SQL queries (Phase B) — all faction relations, recent events, active wars, player
   standings, world knowledge
3. Sends the context to the LLM via `intel_political_dm.prompt`
4. Processes the LLM's response: validates the chosen event, adjusts relation scores, creates war declarations if
   thresholds crossed, injects awareness facts onto NPC leaders, spawns physical manifestations (battles,
   assassinations, brawls)

**War Mechanics:**

- Wars are declared when faction-pair relation drops below the war threshold (default -50)
- Each war tracks morale (0–100) and strength for both sides
- Morale decays each tick; battles adjust morale and strength
- Wars end via surrender (morale ≤ 0), peace deal (LLM event), or forced resolution
- Player participation in battles grants significant standing bonuses/penalties

### 5.14 Battle Manager

**Files:** `BattleManager.cpp`, `BattleManager.h`

Manages both off-screen and player-witnessed faction battles:

**Pending Battles:**
When the Political DM generates a `battle_result` event, a pending battle is placed at the event's location. A 3-second
proximity poll checks if the player comes within 5000 units. If the player arrives, the battle spawns visually;
otherwise, it resolves off-screen.

**Active Battle Lifecycle:**

1. **Spawn** — Soldiers appear at calculated positions (faction A on one side, faction B on the other, 400 units apart)
2. **Combat Init** — Soldiers are paired and forced into combat via `StartCombat()`
3. **Wave Reinforcements** — Up to 5 waves (6+5+4+4+3 soldiers per side) spawn behind existing lines
4. **Morale Tracking** — Deaths reduce morale; player kills grant larger morale swings
5. **Player Participation** — Auto-join if standing ≥ 20 with one side; grants ally/enemy standing changes
6. **Battle End** — Triggered by morale collapse or total elimination
7. **Finalization** — Record result in PoliticalDB, apply standing changes, inject witness facts, generate narrator
   dialogue
8. **Cleanup** — Progressive soldier despawn by distance from player

**Crime Handling:**
During battles, player bounty factions are temporarily removed (preventing bounty spam from friendly-fire) and restored
after cleanup.

### 5.15 Dialogue Tracker

**Files:** `DialogueTracker.cpp`, `DialogueTracker.h`

Counts dialogue lines per NPC by listening to SkyrimNet's dialogue callback. When the count exceeds a configurable
threshold, fires a `dynamic_bio_update` ModEvent telling SkyrimNet to refresh the NPC's bio template. This keeps NPC
self-descriptions evolving as conversations accumulate.

### 5.16 Quest State Tracker

**Files:** `QuestStateTracker.h`

Holds metadata about the currently active IntelEngine quest (if any) — location, quest subtype, enemy type, quest
giver, briefing, victim, item, faction. This data is served via a SkyrimNet decorator so NPCs are aware of active
quests in their bio context.

### 5.17 Dashboard UI Manager

**Files:** `DashboardUIManager.cpp/h`, `DashboardConfig.cpp/h`

Manages the in-game React dashboard via PrismaUI (a web-view overlay framework for Skyrim):

- Creates a PrismaUI view pointing to `dashboard/dist/index.html`
- Registers a hotkey (configurable via `settings.yaml`) with modifier support (Ctrl/Shift/Alt)
- Polls `GetAsyncKeyState` on a background thread for hotkey detection
- Pushes state updates to the dashboard via `window.updateFullState(json)` JavaScript interop
- Receives action callbacks from the dashboard (cancel task, toggle story type, change setting, dispatch action, etc.)
- Gracefully degrades if PrismaUI is not installed

### 5.18 Settings

**Files:** `Settings.cpp`, `Settings.h`

Reads `IntelEngine.ini` for C++-side configuration:

- Debug level, fuzzy match threshold
- Stuck detection: max checks, max recovery attempts
- Departure detection: min checks, max retries
- Story engine: waypoint search radius, min absence days
- MemoryDB: SkyrimNet database path, max memories in context

### 5.19 External API Interfaces

## 6. Papyrus Scripts

### 6.1 IntelEngine.psc — Native API

A `Hidden` global script declaring all native functions as `Global Native`. This is the Papyrus-side API surface —
every function here maps 1:1 to a C++ implementation in `Papyrus.cpp`. Other scripts call these as
`IntelEngine.FindNPCByName(name)` etc.

### 6.2 IntelEngine_Core.psc — Quest Backbone

**Extends:** Quest

The central orchestrator. Manages:

**Slot System:**

- 5 pairs of quest aliases: `AgentAlias[0-4]` (the NPC performing the task) + `TargetAlias[0-4]` (fetch/escort
  targets)
- `AllocateSlot()` finds the first empty slot, forces the NPC into it, and applies `IntelEngine_TaskFaction`
- `ClearSlot()` removes packages, clears faction, resets alias, and starts cooldown
- Slot state constants: 0=empty, 1=traveling, 2=at_destination, 3=returning, 8=at_target, 9=presenting

**Initialization (`Maintenance()`):**
Called on first install and every game load:

1. Registers SkyrimNet decorators (task awareness, political awareness, quest info, schedule info)
2. Recovers active tasks from SlotTracker co-save data → repopulates Papyrus arrays
3. Starts the story engine scheduler
4. Initializes the schedule monitor
5. Initializes politics if enabled

**AI Package Management:**

- `ApplyPackage()` applies one of the defined AI packages (walk/jog/run/sandbox) to an alias
- `RemoveIntelPackages()` strips IntelEngine packages but preserves vanilla AI
- Package priorities: Travel (100) > At-Destination Sandbox (90) > Default (0)

**SkyrimNet Integration:**

- `SendTaskNarration()` — Injects a transient narrative fact into the NPC's SkyrimNet bio
- `SendPersistentMemory()` — Stores a permanent memory via SkyrimNet
- `SendTransientEvent()` — Fires a world event visible to nearby NPCs
- `InjectFact()` — Stores a bio fact keyed by category for NPC awareness

### 6.3 IntelEngine_Travel.psc — Navigation

**Extends:** Quest

Handles NPC travel to destinations with a multi-phase state machine:

**Travel Flow:**

1. `GoToLocation()` — Resolves destination via C++ `ResolveAnyDestination()`, calculates deadline, applies walk/jog/run
   package with linked ref targeting
2. **Departure Check** — Monitors whether the NPC actually started moving (C++ DepartureDetector)
3. **Stuck Recovery** — If stuck, escalates: soft recovery (re-evaluate AI) → waypoint navigation → progressive
   teleport → timeout
4. **Arrival Detection** — C++ ProximityMonitor (150ms) or Papyrus OnUpdate (0.5s fallback) detects when the NPC
   reaches the destination
5. **Linger System** — Three-phase post-arrival behavior:
   - Phase 1: Approach player, greet if present
   - Phase 2: Sandbox briefly at location
   - Phase 3: Release (return to normal AI or proceed with task follow-up)

**Meeting Support:**
When a scheduled meeting triggers travel, `OnMeetingPlayer()` handles the rendezvous — including late arrival
detection, no-show handling, and grace period management.

### 6.4 IntelEngine_NPCTasks.psc — NPC Interactions

**Extends:** Quest

Implements multi-step NPC tasks:

- **FetchNPC** — Agent travels to target, forces target into TargetAlias, returns with target. Three states: traveling
  → at target → returning with target.
- **DeliverMessage** — Agent travels to recipient, delivers message (stored via `StoreReceivedMessage()` on recipient's
  SkyrimNet bio for persistent awareness), optionally schedules a follow-up meeting.
- **SearchForActor** — Player-led search where the agent follows the player. If the player gets too far ahead (>1500
  units), the agent waits.
- **EscortTarget** — Agent fetches target NPC, then escorts both player and target to destination. Three phases: fetch
  → escort → linger at destination.

### 6.5 IntelEngine_Schedule.psc — Time-Based Dispatch

**Extends:** Quest

Manages up to 10 scheduled tasks that trigger at specific game times:

**Schedule Types:**

- `ScheduleMeeting(npc, destination, timeCondition)` — NPC travels to a location at a specified time
- `ScheduleFetch(npc, targetName, timeCondition)` — NPC fetches someone at a scheduled time
- `ScheduleDelivery(npc, targetName, msgContent, timeCondition, meetLoc, meetTime)` — Message delivery at scheduled
  time, with optional meeting follow-up

**Time Parsing:**
Natural language time conditions ("at dawn", "tonight", "in 3 hours", "tomorrow evening") are parsed by C++
`ParseTimeCondition()` into absolute game time values.

**Early Departure:**
The system calculates estimated travel time using `CalculateDeadlineFromDistance()` and dispatches the NPC early enough
to arrive on time (with a configurable buffer, default 2 hours, capped to 75% of the time until the meeting).

**Monitoring:**
Hybrid polling: `OnUpdateGameTime()` for distant future tasks + `OnUpdate()` for imminent tasks (< 1 game hour away).
State tracking: 0=pending, 1=dispatched, 2=meeting active.

### 6.6 IntelEngine_StoryEngine.psc — Story DM

**Extends:** Quest

The autonomous story generation system. Two parallel subsystems:

**Story DM (Player-Centric):**
Runs on a configurable interval (default: periodic tick). Each tick:

1. `BeginAsyncStoryDMTick()` — C++ builds context on a worker thread (Phase A snapshot → Phase B SQL)
2. `OnStoryDMContextReady()` — Context arrives; sends to LLM via `intel_story_dm.prompt`
3. `OnStoryDMResponse()` — Parses JSON response; dispatches the chosen story type

**Story Types:**

- **`seek_player`** — NPC travels to find the player (absence-driven, intimacy-driven, unresolved memory).
- **`informant`** — NPC approaches with gossip about another NPC (traceable to real memories).
- **`road_encounter`** — NPC crosses paths with the player while traveling to their own destination.
- **`ambush`** — Hostile NPC stalks and attacks the player (requires real grudge + combat capability).
- **`faction_ambush`** — Faction soldiers waylay the player (3–7 soldiers, scales with hostility).
- **`stalker`** — Obsessive/romantic NPC follows the player (sneaks, flees when caught, no combat).
- **`message`** — NPC delivers a verbal message (optional meeting invitation).
- **`quest`** — NPC requests player help — subtypes: combat, rescue, find_item, faction_combat,
  faction_rescue, faction_battle.
- **`npc_interaction`** — Two NPCs interact independently (argument, deal, romance, training, favor,
  warning).
- **`npc_gossip`** — Rumors spread through NPC chains (up to 10 people, traceable).

**Quest System:**
Quest subtypes spawn enemies, chests with items, bosses, or rescue victims at dungeon locations.
`FindDeeperSpawnPoint()` scans for dungeon landmarks (word walls, boss chests, coffins) to place objectives deeper in
dungeons rather than at entrances.

**NPC Social Tick:**
A separate tick evaluates loaded NPC groups and generates NPC-to-NPC interactions. Uses `intel_story_npc_dm.prompt` to
decide interactions grounded in real shared memories.

### 6.7 IntelEngine_Politics.psc — Faction System

**Extends:** Quest

Orchestrates the political simulation tick:

1. `TickNow()` — Self-gates on configured interval (default 6 game hours)
2. `ProcessActiveWars()` — C++ handles morale decay on active wars
3. `BeginAsyncPoliticalTick()` — C++ builds political context asynchronously
4. `OnPoliticsDMContextReady()` — Sends context to LLM via `intel_political_dm.prompt`
5. `OnPoliticalDMResponse()` — C++ processes: validates event, adjusts relations, declares wars if threshold crossed,
   applies player standing changes
6. `InjectPoliticalFact()` — Stores political awareness facts on loaded faction leaders' SkyrimNet bios
7. Triggers battle spawning or political event manifestation if applicable

**Player Conduct Reporting:**
NPCs can report player conduct to their faction via `intel_report_player_conduct` action (triggered when the player
expresses political opinions in dialogue). Sentiment levels: strongly_positive (+8), positive (+4), negative (-4),
strongly_negative (-8).

### 6.8 IntelEngine_Battle.psc — Warfare

**Extends:** Quest

Spawns and manages faction battles when the player is nearby:

**Battle Lifecycle:**

1. **Schedule** — Pending battle placed at location with game-time trigger
2. **Proximity Detection** — 3-second poll checks if player is within 5000 units
3. **Spawn Sequence** — `ExecuteFullBattleSpawn()` in C++ handles: position calculation, soldier spawning (leveled
   lists), faction assignment, combat pairing, player auto-join evaluation
4. **Wave Management** — 5 waves of reinforcements (6/5/4/4/3 soldiers per side)
5. **Poll Loop** — Every 3 seconds: check deaths, update morale, trigger wave advances, detect battle end
6. **Finalization** — C++ `FinalizeBattle()` applies standing changes, records events, builds narrative
7. **Cleanup** — Progressive soldier despawn based on distance from player; guard restoration

**Standing Changes from Battles:**

- Kill enemy soldier: -2 standing with their faction per kill
- Allied victory: +15 with ally, -10 with enemy
- Allied defeat: +5 with ally (sympathy), -5 with enemy
- Player participation amplifies all deltas

**Event Manifestation:**
Smaller political events (brawl, border_skirmish, assassination_attempt) are physically manifested when the player is
nearby — spawning NPCs who act out the event.

### 6.9 IntelEngine_PlayerAlias.psc — Bootstrap

**Extends:** ReferenceAlias (filled with the player)

Minimal auto-initialization script:

- `OnInit()` — First install: starts the quest, calls `Core.Maintenance(true)`
- `OnPlayerLoadGame()` — Every load: ensures quest running, calls `Core.Maintenance(false)`

This eliminates the need for an SEQ file and ensures the mod initializes reliably.

### 6.10 IntelEngine_MCM.psc — Settings Menu

**Extends:** SKI_ConfigBase (SkyUI MCM framework)

SkyUI-based configuration with pages for:

- **Active Tasks** — 5 slot status display with cancel buttons
- **Scheduled Tasks** — 10 schedule slots with cancel buttons
- **Settings** — Debug mode, max tasks, travel confirmation policy, stuck detection timeout
- **Story Engine** — Story type toggles, candidate policies, quest type toggles
- **Politics** — Faction relation display, war status
- **Active Packages** — Lists loaded actors running IntelEngine AI packages

## 10. Persistence & Save Safety

The system uses multiple persistence layers, each chosen for its strengths:

- **SKSE Co-Save** — SlotTracker (5 task slots), per-save unique ID. *Every game save.*
- **SQLite (PoliticalDB)** — Faction relations, events, wars, battles, player standings. *On every political
  event.*
- **StorageUtil (Papyrus)** — Per-NPC data: received messages, schedule state, meeting outcomes. *During
  gameplay.*
- **Papyrus Quest Properties** — Slot arrays, schedule arrays, story state, MCM settings. *Automatic with
  save.*
- **GlobalVariables** — MCM toggle settings (debug, max tasks, intervals). *Automatic with save.*
- **SkyrimNet Bio Facts** — NPC awareness data (tasks, politics, gossip). *During gameplay.*
- **JSON State File** — `political_state.json` for immediate NPC awareness. *After each political tick.*
- **YAML Config Files** — Faction definitions, dashboard settings, action toggles. *Persistent on disk.*

**Save-Load Recovery Sequence:**

1. SKSE fires `kPreLoadGame` → Reset all runtime singletons
2. SKSE fires `LoadCallback` → Deserialize SlotTracker from co-save
3. SKSE fires `kPostLoadGame` → Initialize PoliticalDB (with timeline cleanup) → Resolve quest handle → Fixup script
   properties → Dispatch `Maintenance()`
4. `Maintenance()` in Papyrus → Sync slot arrays from C++ co-save data → Register decorators → Restart story engine →
   Resume schedules

## 11. Key Architectural Patterns

### Three-Phase Async Pattern

Used by Story DM, NPC Social, and Politics ticks to keep SQL off the game thread:

- **Phase A** (main thread): Capture engine state into value-only structs
- **Phase B** (worker thread): SQL queries + markdown formatting (no engine API calls)
- **Phase C** (main thread): Dispatch result to Papyrus VM

### Fuzzy Search Cascade

Applied by NPCIndex, ItemIndex, LocationResolver:

1. Exact match (case-insensitive)
2. Article-stripped match ("the Bannered Mare" = "Bannered Mare")
3. Token overlap scoring (word-by-word)
4. Levenshtein distance (character-level typo tolerance)

### Escalating Recovery

Applied to StuckDetector and DepartureDetector:

- Level 1: Position polling detects lack of progress
- Level 2: Soft recovery (re-evaluate AI packages, try waypoint navigation)
- Level 3: Progressive teleport closer to target (2000 → 1000 → 500 → 250 units)

### Soft Dependency Loading

SkyrimNet and PrismaUI are loaded via `GetProcAddress` at runtime. If either DLL is absent, all function pointers
remain null, and calling code checks for null before invocation. The plugin continues with reduced functionality.

### Candidate Pool Isolation

Each DM tick rebuilds its candidate pool from scratch. Async DM responses resolve NPC names against the pool that was
active when the prompt was sent, not the current pool. This prevents save-load mid-tick from dispatching to the wrong
NPC.

### Dispatch History Ring Buffer

The last 6 dispatches (type, NPC, narration, outcome: dispatched/rejected) are maintained in a fixed-size ring buffer
and injected into the DM prompt. This lets the LLM see its own recent decisions, avoid retry loops on rejected
dispatches, and maintain narrative variety.

### Single Source of Truth

- SlotTracker (C++) is authoritative for task state, not Papyrus StorageUtil
- `BuildBusyReason()` is the single source for SkyrimNet busy-status strings
- PoliticalDB is authoritative for faction relations, not in-memory caches
